## Architecture (as mapped during the audit)

mnemo is a single-process HTTP MCP daemon (module github.com/marcelocantos/mnemo). `main.go` parses flags/subcommands, then `runServe` builds a `registry.Registry` (one `store.Store` per `?user=` identity), an `mcp-go` streamable-HTTP server mounted at `/mcp`, a REST/dashboard mux, an SSE event hub, a diagnostics scheduler, an optional federation mTLS listener, and an optional menu-bar shim supervisor. Each per-user `Store` opens SQLite twice: `writeDB` with `SetMaxOpenConns(1)` (Go pool serialises all writers onto one connection) and `readDB` with the default unbounded pool ("many concurrent readers"). Both use WAL, `synchronous=NORMAL`, `busy_timeout=5000`, mmap. Schema is applied before opening long-lived handles: a fresh/empty DB runs `schema.sql` directly (bypassing sqlift), an existing DB is migrated by sqlift under `ApplyOptions{}` (== AllowNone) with a pre-migration VACUUM INTO backup; a rejected migration is logged and the daemon proceeds in degraded read-only-compatible mode (the old wipe-and-reindex path is gone).

Threading/concurrency model: per user, `registry.startWorkers` spawns (tracked by `sync.WaitGroup`s and cancelled via `r.baseCtx`): a startup ingest goroutine that runs `IngestAll` then all the `Ingest*` source indexers then `store.Watch()` (an fsnotify loop that debounces events at 300 ms and dispatches `ingestFile`/`ingestCodexFile`/source-file ingest via `time.AfterFunc` callbacks — each callback runs on its own goroutine but writes serialise on the single write connection), image describer/OCR/embedder workers, the compaction watcher (`compact.NewWatcher(...).Run`), the CLAUDE.md reviewer, a 6-hour `PRAGMA optimize` ticker, the connection sweeper (`MarkStaleConnectionsClosed`), the daily backup worker, and (opt-in) the Anthropic Admin reconciler. `ingestFile` holds `s.mu` only to read/write the in-memory `offsets` map; DB work happens inside a `writerState` transaction that commits every ~200 ms.

Critical gap in the shutdown/threading model: on non-Windows platforms there is NO signal handling. `main` calls `runServe(context.Background(), ...)`, and that same background context is threaded into `registry.NewRegistry` as `baseCtx`. The `select { case <-ctx.Done(): ... }` graceful-drain block in `runServe` (which calls `httpSrv.Shutdown`, closes the federated server, and would let `defer reg.Close()` run) is therefore unreachable outside the Windows SCM path — SIGTERM/SIGINT just kill the process, so background workers are never drained and no explicit WAL checkpoint is issued (T97.1). `store.Close` itself also issues no `PRAGMA wal_checkpoint(TRUNCATE)`, relying on SQLite's last-connection-close auto-checkpoint, which never runs on an abrupt Unix kill.

The MCP query surface: `mnemo_query`/`mnemo_evaluate` → `callHandler.query` → `Store.Query`, which unconditionally passes input through `sqldeep.Transpile`, grabs a `readDB.Conn`, sets `PRAGMA query_only=1`, runs the statement, caps at 100 rows, and resets `query_only=0` on release. Egress is per-feature gated: the reconciler requires both `cost_reconciliation.enabled` AND `ANTHROPIC_ADMIN_API_KEY`; federation requires non-empty `linked_instances`; GitHub/CI backfill runs for repos already seen in history using local `gh` auth.

## Load-bearing invariants

- Schema is append-only: sqlift must ALWAYS be invoked with AllowNone (store.go:666 `sqlift.ApplyOptions{Allow: sqlift.AllowNone}`) — no gate (AllowRebuild/AllowDestructive/AllowLoosen/AllowDataDependent) may ever be enabled, or an older binary reading the DB can crash or lose data.
- The removed wipe-and-reindex path must never resurrect: a schema-version mismatch must degrade to read-compatible mode (New() logs and proceeds), never delete mnemo.db — some users' source JSONL has been pruned by Claude Code, so a wipe is permanent data loss.
- mnemo_query must remain read-only: PRAGMA query_only=1 must be set before every statement and reset only after; no code path may run user SQL on a connection where query_only is off, and the surface must not permit writes via triggers/CTE/ATTACH or privilege escalation.
- External egress is opt-in: the Anthropic Admin reconciler fires only when BOTH config.enabled AND ANTHROPIC_ADMIN_API_KEY hold; federation fires only when linked_instances is non-empty. Restoring an older backup must not silently re-enable any of these.
- Ingest offset monotonicity: the persisted ingest_state.offset must never advance past the last fully-processed line, or resumed/crash-recovered ingest permanently skips transcript content.
- The fresh-schema fast path (applyFreshSchema executing schema.sql directly) must produce byte-for-byte the same schema sqlift would build from schema.sql, since it bypasses sqlift's diff entirely.
- All writes go through the single writeDB connection (SetMaxOpenConns(1)); no code may open a second writer handle expecting concurrent write access.
- Entry dedup relies on INSERT OR IGNORE over (session_id, uuid); message/content extraction is gated on entryID!=0 to avoid duplicate content rows on re-ingest.

## Trust boundaries

- JSONL transcript files under ~/.claude/projects/ (jsonlEntry parsing in store.go ingestFile) and ~/.codex/sessions/ rollout files (codex.go) — externally-written, possibly partial/corrupt/oversized content parsed line-by-line.
- mnemo_query `query` argument from any MCP client — arbitrary SQL/sqldeep string executed against the DB (tools.go query handler → Store.Query).
- The ?user=<name> HTTP query parameter (UsernameContextFunc) — selects/creates a per-user Store and drives home-dir path resolution.
- ~/.mnemo/config.json — workspace_roots, extra_project_dirs, synthesis_roots, vault_path, linked_instances peer certs, cost_reconciliation flag; parsed at startup and via mnemo_config hot-reload.

- mnemo_note_post/recv inbox directory paths (canonicalisation of ~, ./.., symlinks) supplied by callers.
- Responses from the Anthropic Admin API (reconciler.go costReportResponse) and the GitHub API / gh CLI (PR/issue/CI backfill).
- Image data extracted from transcripts (base64/file-path) decoded and stored as BLOBs.

### F1 — [CRITICAL] Oversized JSONL line silently aborts ingest and permanently advances the offset past unread content (scanner.Err() never checked at any ingest site)

- **Location:** `internal/store/store.go:3175`
- **Dimension:** -
- **Invariant violated:** Ingest offset monotonicity (recon invariant #5): the persisted ingest_state.offset must never advance past the last fully-processed line, or resumed/crash-recovered ingest permanently skips transcript content. Also: 'The ingest path is fully local... Indexes all content block types' — content must not be silently dropped.
- **Failure scenario:** A completed Claude session transcript contains one message with an inline base64 screenshot, making that JSONL line ~1.5 MB (>1<<20 buffer cap). At startup IngestAll -> parseFile scans from offset 0; when it reaches that line bufio.Scanner returns false with err=bufio.ErrTooLong. The loop exits as if EOF, scanner.Err() is never inspected, and `pf.newOffset, _ = f.Seek(0, 1)` records a position ~1 MB INTO the middle of the oversized line (my repro: fileSize=2097221, persisted newOffset=1048598). parseFile returns (pf, nil), so runWriter treats it as success and writeParsedFile (store.go:3067) persists that advanced offset to ingest_state. Result: the oversized message is never inserted into `entries` or `messages` (mnemo_search / mnemo_read_session never return it), and every message after it in the session is also absent. Because IngestAll only re-reads when offset < size and this is a static completed file the watcher never re-fires, the tail stays missing and the oversized message is lost forever. No error is logged. The identical bug exists in the watcher path (ingestFile: loop ends at store.go:6064 with no scanner.Err() check, then the final `newOffset, _ := f.Seek(0,1)` at store.go:6082-6085 advances ingest_state) and in the Codex path (parseCodexFile: loop store.go/codex.go:145, no scanner.Err() check, `pf.newOffset, _ = f.Seek(0, 1)` at codex.go:221; cap is 8<<20 but the code comment itself says 'Codex tool outputs can be large').
- **Evidence:**

```
for scanner.Scan() {
		line := scanner.Bytes()
		...
	}

	pf.newOffset, _ = f.Seek(0, 1)   // <- store.go:3175: no scanner.Err() check; on ErrTooLong this is mid-oversized-line
	return pf, nil                    // returns nil error, so writeParsedFile persists the advanced offset

// ingestFile mirror (store.go:6082-6085), also no scanner.Err() after the loop:
	newOffset, _ := f.Seek(0, 1)
	ws.tx.Exec(`INSERT OR REPLACE INTO ingest_state (path, offset, recorded_size, recorded_mtime) VALUES (?, ?, ?, ?)`, path, newOffset, recSize, recMtime)

// codex parseCodexFile mirror (codex.go:221):
	pf.newOffset, _ = f.Seek(0, 1)
```

- **Fix sketch:** Stop advancing the offset when the scan aborts abnormally. After each scan loop add `if err := scanner.Err(); err != nil { return ... }` (parseFile/parseCodexFile return the error so pf is discarded and the offset is NOT persisted; ingestFile returns before its final ingest_state write). To actually ingest oversized lines rather than loop forever on them, replace bufio.Scanner with a bufio.Reader + ReadBytes('\n') (no per-line cap), streaming the raw bytes; or on ErrTooLong read forward to the next '\n' with an uncapped reader so exactly one line is skipped and following lines still ingest. Whatever the choice, the persisted offset must never point past a line that was not inserted.
- **Verification:** Before: create a project transcript file with a small line, then a >1MB line (a message with `"type":"image"` base64 data), then two more small lines; run `bin/mnemo` once; query `SELECT offset FROM ingest_state WHERE path=...` (offset lands mid-oversized-line, < file size) and `SELECT count(*),max(rowid) FROM entries WHERE session_id=...` (missing the oversized line and everything after); grep logs — no error emitted. The mechanism is reproduced directly by scratchpad/audit-repro/mnemo/repro.go (prints `scanner.Err() = bufio.Scanner: token too long`, processed=[line1], recovered-on-reingest=[]). After: the oversized line and all following lines appear in `entries`/`messages`, or at minimum ingest_state.offset stays at the start of the un-ingested region and the error is logged.

### F2 — [CRITICAL] mnemo_query read-only guard bypassable: `PRAGMA query_only=0; <write>` executes writes (DB wipe/corruption) — also reachable over federation

- **Location:** `internal/store/store.go:5535`
- **Dimension:** -
- **Invariant violated:** mnemo_query must remain read-only: query_only=1 must hold for every statement; no code path may run user SQL on a connection where query_only is off, and the surface must not permit writes via triggers/CTE/ATTACH or privilege escalation (recon invariant #3; federated.go contract 'peers cannot exercise any write or control surface').
- **Failure scenario:** An MCP client (or an mTLS federation peer — `mnemo_query` is in FederatedToolNames, internal/tools/federated.go:29) calls mnemo_query with `PRAGMA query_only=0; DELETE FROM messages`. sqldeep.Transpile passes it through unchanged (`PRAGMA query_only = 0; DELETE FROM messages`). mattn/go-sqlite3's QueryContext executes ALL statements on the SAME connection: the leading PRAGMA re-enables writes, then the DELETE runs and commits. Verified: messages row count went 3 -> 0 with err=nil. Any INSERT/UPDATE/DROP/DELETE (including wiping the whole transcript DB, which per the schema-policy invariant is permanent data loss because source JSONL may be pruned) is thus executable through the 'read-only' surface, and remotely via a linked federation instance whose only containment is this guard.
- **Evidence:**

```
if _, err := conn.ExecContext(ctx, "PRAGMA query_only = 1"); err != nil {
		return nil, fmt.Errorf("set query_only: %w", err)
	}

	rows, err := conn.QueryContext(ctx, execSQL, args...)
```

- **Fix sketch:** Do not rely on a resettable per-connection PRAGMA for a hard security boundary. (1) Open the read pool with SQLITE_OPEN_READONLY (DSN `file:...?mode=ro`) so writes are refused at the VFS/file-handle level and cannot be undone by any PRAGMA. (2) Install an sqlite3 authorizer (mattn ConnectHook -> conn.RegisterAuthorizer/RegisterFunc equivalent) that returns SQLITE_DENY for SQLITE_ATTACH and SQLITE_PRAGMA. (3) Reject multi-statement input in Store.Query (parse/verify exactly one statement) before execution, since QueryContext silently runs trailing statements. Keep query_only=1 as belt-and-suspenders.
- **Verification:** Repro at /private/tmp/claude-501/-Users-marcelo-think/d9b0c9f7-c002-4328-aa22-81d64543a94d/scratchpad/audit-repro/mnemo (main.go) reproduces the exact pipeline. Before fix: `go run -tags sqlite_fts5 .` prints '[after test1] messages count = 0' (write succeeded). After fix: TEST 1 must return an error and count must stay 3. Equivalent daemon check: call mnemo_query with `PRAGMA query_only=0; UPDATE session_meta SET repo='x'` and confirm it is rejected and no row changes.

### F3 — [HIGH] triggerImageSidecars spawns unbounded goroutines that pin full image bytes; semaphore gates execution, not spawn count → goroutine/memory blowup on every startup

- **Location:** `internal/store/images.go:203`
- **Dimension:** -
- **Invariant violated:** Background image processing must be bounded in both concurrent work AND in the number of queued goroutines and the memory they retain; the imageSem semaphore was introduced to cap image-sidecar concurrency but it caps only execution, not fan-out.
- **Failure scenario:** backfillImages (internal/store/store.go:2866) runs at the end of IngestAll on every daemon start. It loads every entry containing an inline image into the `entries` slice, then loops synchronously calling ingestImagesForEntry -> triggerImageSidecars for each. triggerImageSidecars (images.go:199-206) fires THREE goroutines per image (go s.runSidecar(ocr), go s.runSidecar(describe), go s.runSidecar(embed)) BEFORE any of them acquires imageSem. runSidecar (images.go:210-214) acquires `s.imageSem <- struct{}{}` only *inside* the goroutine. Each goroutine closure captures `data []byte` — the full decoded image (describeOneImage(s.writeDB, imageID, data, mimeType), etc.). For a user with, say, 10,000 images totalling ~2 GB, backfillImages spawns ~30,000 goroutines that all block on a NumCPU-sized (e.g. 16-slot) semaphore, each holding its decoded image bytes live until it is scheduled. Result: ~30k goroutines and ~2 GB of decoded image data pinned simultaneously on every startup, on top of the raw-JSON `entries`/`msgs` slices already holding the whole set — a transient goroutine + memory spike proportional to total image count that can OOM image-heavy installs. It happens even when all sidecar work is already done, because the idempotency short-circuit (e.g. describeOneImage's LookPath + INSERT OR IGNORE no-op) runs only after the goroutine wins a semaphore slot. The Watch path multiplies this: each ingestFile spawns its own fire-and-forget goroutine (store.go:6107) that queries up to 100 image entries and calls triggerImageSidecars for each (up to 300 goroutines per file), and concurrent debounced ingests stack these with no aggregate cap.
- **Evidence:**

```
go s.runSidecar(func() { ocrOneImage(s.writeDB, imageID, data) })
	go s.runSidecar(func() { describeOneImage(s.writeDB, imageID, data, mimeType) })
	go s.runSidecar(func() { embedOneImage(s.writeDB, imageID, data, mimeType) })
// ...
func (s *Store) runSidecar(fn func()) {
	s.imageSem <- struct{}{}
	defer func() { <-s.imageSem }()
	fn()
}
```

- **Fix sketch:** Acquire the semaphore at the call site before spawning, so imageSem bounds goroutine count and pinned memory, not just execution. E.g. change triggerImageSidecars to `for _, fn := range sidecars { s.imageSem <- struct{}{}; go func(fn func()){ defer func(){ <-s.imageSem }(); fn() }(fn) }` (this makes backfillImages block/back-pressure as it feeds). Additionally, stream backfillImages row-by-row instead of buffering every image entry into a slice, and track the sidecar/6107 goroutines in a WaitGroup so Store.Close does not race a closing DB.
- **Verification:** Before: seed a test Store with N images having pending sidecar work, set imageSem capacity to 1, call backfillImages and sample runtime.NumGoroutine() (or a spawn counter) at the peak — observe ~3N goroutines and N*imageSize bytes retained (pprof heap/goroutine profile). After the fix: peak goroutine count stays <= NumCPU + O(1) and retained image bytes stay <= NumCPU * maxImageSize regardless of N.

### F4 — [HIGH] ATTACH DATABASE is permitted under query_only=1: reads arbitrary local SQLite files (info disclosure) and creates files at attacker-chosen paths

- **Location:** `internal/store/store.go:5531`
- **Dimension:** -
- **Invariant violated:** mnemo_query must not permit reads outside the transcript DB or any filesystem side-effects; the read-only surface 'must not permit writes via triggers/CTE/ATTACH or privilege escalation' (recon invariant #3).
- **Failure scenario:** PRAGMA query_only blocks writes to the schema but does NOT block ATTACH. A client/federation peer calls mnemo_query with `ATTACH DATABASE '/absolute/path/other.db' AS x; SELECT * FROM x.<table>` and reads any SQLite database readable by the daemon user — e.g. another user's ~/.mnemo/mnemo.db in multi-user mode, browser history, or app databases. Verified: attaching a secret DB and selecting returned `{v:TOPSECRET}` with err=nil. Separately, `ATTACH DATABASE '/path/new.db' AS y` on a non-existent path CREATES an empty SQLite file on disk even under query_only=1 (verified: '!! FILE CREATED'), i.e. an arbitrary-path filesystem write through the read-only surface. Both are exposed over the federated mTLS endpoint.
- **Evidence:**

```
if _, err := conn.ExecContext(ctx, "PRAGMA query_only = 1"); err != nil {
		return nil, fmt.Errorf("set query_only: %w", err)
	}

	rows, err := conn.QueryContext(ctx, execSQL, args...)
```

- **Fix sketch:** Deny ATTACH: install an sqlite3 authorizer returning SQLITE_DENY for the SQLITE_ATTACH action (and SQLITE_DETACH), or set SQLITE_LIMIT_ATTACHED to 0 on the read connection. Opening the read pool with mode=ro does not by itself stop ATTACH-for-read, so the authorizer/limit is required. Add a statement pre-check rejecting ATTACH/DETACH tokens as defense-in-depth.
- **Verification:** Same repro (TEST 2 / TEST 3): before fix, `ATTACH ...; SELECT v FROM x.s` returns TOPSECRET and TEST 3 creates a file; after fix both must error and no file is created. Daemon check: `mnemo_query` with `ATTACH DATABASE '/etc/hosts' AS z` must be rejected.

### F5 — [HIGH] Mid-stream commit persists f.Seek(0,1) which overshoots the last processed line, so a crash/kill between a mid-stream commit and completion permanently skips buffered-but-unwritten lines

- **Location:** `internal/store/store.go:6042`
- **Dimension:** -
- **Invariant violated:** Ingest offset monotonicity (recon invariant #5): the persisted ingest_state.offset must never advance past the last fully-processed line. bufio.Scanner reads ahead of the last line returned by Scan(), so f.Seek(0,1) does not reflect the last line actually inserted into the DB.
- **Failure scenario:** The watcher's ingestFile processes a file with a sizeable unindexed tail (e.g. mnemo was stopped while an active agent session appended hundreds of lines; the debounced ingestFile then processes that whole tail from the stored offset). On the first `scanner.Scan()` bufio fills its buffer by reading up to ~1 MB from the file, so after only a handful of lines have been returned/inserted, `f.Seek(0,1)` already points ~1 MB into the file (repro2.go: after processing 1 line, f.Seek(0,1)=12300, which spans all 300 lines of the test file). When the mid-stream commit fires (>=50 lines AND >=200ms), store.go:6042 records `curOffset := f.Seek(0,1)` and commits it to ingest_state and s.offsets[path], while the transaction only contains the lines actually returned by Scan() so far. Per the recon, Unix mnemo has no signal handling and is terminated by SIGTERM/SIGKILL (e.g. `brew services restart mnemo`, reboot) with no graceful drain. If the process dies after this mid-stream commit but before the final commit for the rest of the buffered region, restart's IngestAll resumes from curOffset (loaded at store.go:810-817), skipping every line between the last DB-inserted line and curOffset. Those lines are never re-read (offset only moves forward) -> permanent loss of a ~1 MB window of transcript content.
- **Evidence:**

```
// store.go:6040-6054 (mid-stream commit inside `for scanner.Scan()`):
			if time.Since(lastCommit) >= commitInterval {
				// Commit current transaction with offset update.
				curOffset, _ := f.Seek(0, 1)   // <- reads AHEAD of last Scan()ed line
				recSize, recMtime := statFingerprint(path)
				ws.tx.Exec(`INSERT OR REPLACE INTO ingest_state (path, offset, recorded_size, recorded_mtime)
					VALUES (?, ?, ?, ?)`, path, curOffset, recSize, recMtime)
				ws.Close()
				if err := ws.tx.Commit(); err != nil { return err }
				s.mu.Lock()
				s.offsets[path] = curOffset
				s.mu.Unlock()
```

- **Fix sketch:** Do not derive the persisted offset from f.Seek(0,1). Track it as a running byte total of lines actually processed, exactly as codex.go already does with `lineStart`/`thisStart`: initialise `consumed := offset`, and after each successfully handled Scan() do `consumed += int64(len(line)) + 1`. Write `consumed` (not f.Seek(0,1)) to ingest_state at both the mid-stream commit (store.go:6042-6045) and the final commit (store.go:6082-6085). This guarantees the committed offset equals the end of the last line inserted in the just-committed transaction. Handle the '\r\n' case if present (the reader strips only '\n').
- **Verification:** Before: add temporary logging (or a test) that, at the mid-stream commit, compares curOffset against the byte end of the highest line inserted into `entries` for that path; observe curOffset >> that value. Or run scratchpad/audit-repro/mnemo/repro2.go which shows f.Seek(0,1)=12300 after 1 processed line (offset spans 300 lines). Reproduce end-to-end: point mnemo at a fresh DB and a large new session file, `kill -9` the daemon shortly after ingest starts, restart, and diff indexed line count vs file line count. After: with offset computed from consumed bytes, the persisted offset never exceeds the last inserted line and the kill/restart cycle loses no lines.

### F6 — [HIGH] No SIGTERM/SIGINT handler on Unix: the entire graceful-drain + WAL-checkpoint shutdown path is dead code (T97/T97.1)

- **Location:** `main.go:189`
- **Dimension:** -
- **Invariant violated:** T97.1: on SIGTERM/SIGINT the daemon must gracefully drain in-flight MCP sessions and background workers and leave a checkpointed WAL; T97: an in-place binary upgrade must not abruptly drop MCP connections.
- **Failure scenario:** mnemo is deployed via `brew services` (per CLAUDE.md). `main` calls `runServe(context.Background(), ...)` (main.go:189) and threads that never-cancelled context into `registry.NewRegistry` as baseCtx. On non-Windows, `runAsServiceIfUnderSCM` is a stub returning (false,nil) (service_other.go:14-18), and there is no `signal.Notify`/`signal.NotifyContext` anywhere in the tree (grep confirms zero matches). So `brew services stop|restart mnemo` and `brew upgrade mnemo` send SIGTERM, which terminates the process immediately: the `case <-ctx.Done()` block that runs `httpSrv.Shutdown`/`httpServer.Shutdown`/`fedSrv.Shutdown` (main.go:831-848) and the `defer reg.Close()` (main.go:554, which would `e.workers.Wait()` and `store.Close()`) never execute. Concrete outcomes on every service stop/restart/upgrade: (1) active Claude Code MCP sessions are severed mid-response instead of drained (clients surface `MCP error -32000: Connection closed`); (2) an in-flight `claude -p` compaction/reviewer subprocess and any running ingest transaction are killed mid-flight; (3) no `PRAGMA wal_checkpoint(TRUNCATE)` is ever issued (store.Close at store.go:825-832 only calls readDB.Close()/writeDB.Close() with no checkpoint, and openDB sets no wal_autocheckpoint override), so the -wal file is left for lazy recovery on next open rather than the clean checkpointed WAL the target requires.
- **Evidence:**

```
if err := runServe(context.Background(), *addr, *federatedAddr); err != nil {   // main.go:189 — ctx never cancels on Unix
// service_other.go:14
func runAsServiceIfUnderSCM(addr, federatedAddr string) (bool, error) { return false, nil }
// store.go:825
func (s *Store) Close() error { rerr := s.readDB.Close(); werr := s.writeDB.Close(); ... }  // no wal_checkpoint
```

- **Fix sketch:** In main, derive the serve context from `ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)` (Unix build) and pass it to runServe so the existing ctx.Done shutdown block and `defer reg.Close()` actually fire. Add `PRAGMA wal_checkpoint(TRUNCATE)` to store.Close before closing the handles (issue on writeDB while readers are already drained by reg.Close's WaitGroup).
- **Verification:** Before: `grep -rn --include=*.go 'signal.Notify' .` returns nothing; start the binary, connect an MCP client, `kill -TERM <pid>`; observe the process exits instantly with no `mnemo serve shutting down` log line and `~/.mnemo/mnemo.db-wal` remains non-zero. After: repeat `kill -TERM`; the shutdown log lines print, httpSrv.Shutdown drains the client cleanly, and `stat -f %z ~/.mnemo/mnemo.db-wal` is 0.

### F7 — [MEDIUM] Debouncer re-entrancy allows two ingestFile/ingestCodexFile goroutines to run concurrently for the same path

- **Location:** `internal/store/debouncer.go:41`
- **Dimension:** -
- **Invariant violated:** The debouncer's documented contract — 'coalesces rapid successive calls for the same key into a single deferred invocation' — plus the ingest design assumption that only one ingest pass per path runs at a time (offsets map + single writeDB connection). Also invariant #8: entry/message dedup must prevent duplicate content rows.
- **Failure scenario:** `enqueue` schedules `time.AfterFunc(window, cb)` where `cb` runs `fn()` FIRST and only then locks and `delete(d.timers,key)` (debouncer.go:41-48). Once the timer has fired and `fn()` (ingestFile) is executing, the map still holds the already-fired timer, so a fresh event during that window calls `t.Stop()` (a no-op on a fired timer) and schedules a SECOND timer for the same key. For a live Claude session whose .jsonl is appended every few seconds while its ingestFile takes >300ms (large backlog / slow disk), the second timer fires and launches a second `s.ingestFile(path)` on its own goroutine concurrently with the first (store.go:3278-3288 dispatch; ingestFile at store.go:5909 has no per-path in-flight guard). Concrete wrong outcomes: (a) both goroutines contend for the single writeDB connection (SetMaxOpenConns(1)) — the redundant duplicate blocks in db.Begin(), stalling ingest of OTHER live sessions' files too, degrading pipeline latency; (b) two committers race `INSERT OR REPLACE INTO ingest_state`, transiently regressing the persisted offset; (c) duplicate rows: `idx_entries_session_uuid` is a PARTIAL unique index (`WHERE uuid IS NOT NULL`, schema.sql:662) and the generated `uuid` is `COALESCE(raw->>'$.uuid', raw->>'$.messageId')`, so any line type without those fields (e.g. `summary`, `file-history-snapshot`, system lines) has uuid NULL, `INSERT OR IGNORE` never dedups it, both concurrent passes get RowsAffected>0, and the `entries` row is inserted twice; for the rarer uuid-less user/assistant entry, the entryID!=0 gate passes for both and `messages` (a plain `INSERT`, no unique constraint, store.go:2887) gets duplicate content rows that show up as duplicate search hits.
- **Evidence:**

```
d.timers[key] = time.AfterFunc(d.window, func() {
	fn()                      // ingestFile runs; still executing here
	d.mu.Lock()
	delete(d.timers, key)     // map entry cleared only AFTER fn() returns
	d.counter++
	d.mu.Unlock()
})
```

- **Fix sketch:** Guard ingest with a per-path in-flight set (map[string]bool under s.mu; skip-or-coalesce if already running, re-enqueue on completion), or restructure the debouncer so the map entry is cleared before fn() runs and a running key is not re-dispatched until it finishes. Simplest: track in-flight paths in the Store and have the debounce callback early-return when the path is already being ingested.
- **Verification:** Before: unit test the debouncer with a slow fn (sleep 500ms) — enqueue key, then enqueue the same key ~50ms later; assert (via an atomic in-flight counter inside fn) that the max concurrent invocations observed is 2, not 1. Integration: append two batches of NULL-uuid `summary` lines to a .jsonl and drive two overlapping ingestFile calls from offset 0; assert `SELECT count(*) FROM entries` doubles. After fix: the debouncer test shows max concurrency 1, and the entries count is stable across overlapping triggers.
